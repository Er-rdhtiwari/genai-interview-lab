# JOI Delivery Repository Reference

## Purpose
This repository is a small FastAPI backend for a delivery platform. The current implementation focuses on grocery ordering and exposes a few HTTP endpoints for:

- adding a product to a cart
- viewing a cart
- checking application health
- checking store inventory health

The application uses in-memory seed data rather than a database, so the code is simple to run and easy to inspect.

## Architecture at a glance
The runtime flow is:

1. `main.py` creates the FastAPI app and initializes seed data.
2. Service instances are attached to `app.state`.
3. Controllers receive HTTP requests and resolve services through FastAPI dependencies.
4. Services operate on domain objects stored in memory.
5. Domain objects serialize themselves through `to_json()` helpers when needed.

The source tree is organized by responsibility:

- `domain/`: business entities
- `service/`: business logic
- `controller/`: HTTP endpoints and request/response models
- `generator/`: seed-data creation
- `system/`: non-business utility routes

## Request flow example
For `POST /cart/product`:

1. FastAPI receives JSON matching `AddProductRequest`.
2. `cart_controller.add_product_to_cart()` resolves `CartService` through `get_cart_service()`.
3. `CartService.add_product_to_cart_for_user()` fetches the user, cart, and product.
4. The service appends the product to the cart and returns a `CartProductInfo` payload.

## Root files

### `app.py`
Minimal startup script. It imports `app` from `joi_delivery.main` and starts Uvicorn on `127.0.0.1:8020` when executed directly.

There are no classes or functions in this file; it is purely a convenience entry script.

## Source package

### `src/joi_delivery/__init__.py`
Package marker for `joi_delivery`. It contains no runtime logic.

### `src/joi_delivery/main.py`
Primary application bootstrap module.

#### Functions

##### `create_app() -> FastAPI`
Builds and configures the FastAPI application.

Responsibilities:

- creates the `FastAPI` instance with title, description, and version metadata
- includes the shared API router from `router.py`
- loads seed data through `initialize_data()`
- creates and stores `UserService`, `ProductService`, and `CartService` on `app.state`
- registers a custom handler for `RequestValidationError`

Important behavior:

- the application is stateful in memory because services are shared through `app.state`
- all startup data comes from the seed generator, not a persistent store

##### `main()`
Starts Uvicorn with the already-created `app` object on port `8020`.

#### Module-level objects

##### `app`
Application instance created eagerly by calling `create_app()`. This is the object imported by tests and by `app.py`.

#### Nested function

##### `custom_validation_exception_handler(request, e)`
Defined inside `create_app()`. It logs validation errors and returns a JSON body with:

- `message`: flattened validation error text
- `data`: `None`

This overrides FastAPI's default validation-error response format.

### `src/joi_delivery/router.py`
Top-level router composition module.

#### Module-level objects

##### `router`
An `APIRouter` that includes:

- cart routes from `controller/cart_controller.py`
- inventory routes from `controller/inventory_controller.py`
- system routes from `system/routes.py`

There are no functions or classes in this file.

### `src/joi_delivery/dependencies.py`
Dependency-provider module for FastAPI controllers.

#### Functions

##### `get_user_service(request: Request) -> UserService`
Returns `request.app.state.user_service`.

##### `get_product_service(request: Request) -> ProductService`
Returns `request.app.state.product_service`.

##### `get_cart_service(request: Request) -> CartService`
Returns `request.app.state.cart_service`.

These helpers centralize service lookup and keep controllers thin.

## Controller layer

### `src/joi_delivery/controller/__init__.py`
Package marker for controller modules. No runtime logic.

### `src/joi_delivery/controller/models.py`
Pydantic request/response models used by HTTP endpoints.

#### Classes

##### `AddProductRequest(BaseModel)`
Request body for adding a product to a cart.

Fields:

- `user_id: str`
- `product_id: str`
- `outlet_id: str`

Each field uses `Field(..., description=...)`, so FastAPI documents them in OpenAPI.

##### `CartProductInfo(BaseModel)`
Response model for the add-to-cart endpoint.

Fields:

- `cart: dict`
- `product: dict`
- `selling_price: float | None = None`

The model accepts serialized dictionaries rather than strongly typed nested models.

### `src/joi_delivery/controller/cart_controller.py`
HTTP endpoints for cart operations.

#### Module-level objects

##### `router`
An `APIRouter` with prefix `/cart` and tag `cart`.

#### Functions

##### `add_product_to_cart(data: AddProductRequest, cart_service: CartService = Depends(get_cart_service))`
Handles `POST /cart/product`.

Behavior:

- validates the request body against `AddProductRequest`
- resolves `CartService` via dependency injection
- calls `cart_service.add_product_to_cart_for_user(data)`
- returns the service result as `CartProductInfo`

##### `view_cart(user_id: str = Query(...), cart_service: CartService = Depends(get_cart_service))`
Handles `GET /cart/view`.

Behavior:

- reads `user_id` from the query string
- calls `cart_service.get_cart_for_user(user_id)`
- returns a `Cart` object directly

Note:
FastAPI is returning a dataclass domain object here, not a Pydantic response model. That works in this small codebase but mixes domain and transport concerns.

### `src/joi_delivery/controller/inventory_controller.py`
HTTP endpoint for inventory health.

#### Module-level objects

##### `router`
An `APIRouter` with prefix `/inventory` and tag `inventory`.

#### Functions

##### `fetch_store_inventory_health(store_id: str = Query(..., alias="store_id", description="Store ID"))`
Handles `GET /inventory/health`.

Current behavior:

- requires a `store_id` query parameter
- ignores the provided store ID
- returns HTTP 200 with an empty body

This is a placeholder endpoint rather than a finished inventory-health feature.

## Service layer

### `src/joi_delivery/service/__init__.py`
Package marker for service modules. No runtime logic.

### `src/joi_delivery/service/user_service.py`
Service for user lookups.

#### Classes

##### `UserService`
Stores a list of `User` objects and exposes simple search behavior.

###### `__init__(self, users: List[User])`
Receives the in-memory user collection and stores it on `self.users`.

###### `fetch_user_by_id(self, user_id: str) -> Optional[User]`
Linear search through `self.users`.

Returns:

- matching `User` if found
- `None` if the ID is unknown

### `src/joi_delivery/service/product_service.py`
Service for product lookup.

#### Classes

##### `ProductService`
Stores a list of `GroceryProduct` objects and resolves products for cart operations.

###### `__init__(self, products: List[GroceryProduct])`
Stores the product collection on `self.products`.

###### `get_product(self, product_id: str, outlet_id: str) -> Optional[GroceryProduct]`
Linear search through all products and returns the first product where:

- `product.product_id == product_id`
- `product.store` exists
- `product.store.outlet_id == outlet_id`

This method ensures the product belongs to the requested outlet.

### `src/joi_delivery/service/cart_service.py`
Primary business-logic service for cart operations.

#### Classes

##### `CartService`
Coordinates user, cart, and product data.

Dependencies:

- `UserService`
- `ProductService`
- `cart_for_users`: dictionary keyed by `user_id`

###### `__init__(self, user_service: UserService, product_service: ProductService, cart_for_users: dict)`
Stores the injected dependencies on the service instance.

###### `add_product_to_cart_for_user(self, add_product_request: AddProductRequest) -> CartProductInfo`
Implements the core add-to-cart flow.

Steps:

1. fetch the user from `UserService`
2. fetch the cart using `fetch_cart_for_user()`
3. fetch the product from `ProductService`
4. append the product to `cart.products`
5. return `CartProductInfo` built from `cart.to_json()` and `product.to_json()`

Behavioral limitations:

- no validation if the user is missing
- no validation if the cart is missing
- no validation if the product is missing
- no quantity handling
- no stock decrement

The method will raise runtime errors if `cart` or `product` is `None`.

###### `get_cart_for_user(self, user_id: str) -> Optional[Cart]`
Fetches the user first, then delegates to `fetch_cart_for_user()`.

###### `fetch_cart_for_user(self, user: User) -> Optional[Cart]`
Returns:

- `None` immediately if `user` is `None`
- the cart from `self.user_carts` using `user.user_id` otherwise

## Domain layer

### `src/joi_delivery/domain/__init__.py`
Package marker for domain modules. No runtime logic.

### `src/joi_delivery/domain/product.py`
Base product abstraction.

#### Classes

##### `Product(ABC)`
Abstract dataclass with shared product fields:

- `product_id: str`
- `product_name: str`
- `mrp: float`

###### `to_json(self)`
Returns a dictionary containing the three shared fields.

Even though `Product` inherits from `ABC`, it does not declare abstract methods, so it behaves mainly as a typed base dataclass.

### `src/joi_delivery/domain/food_product.py`
Restaurant-food specialization.

#### Classes

##### `FoodProduct(Product)`
Placeholder subclass of `Product`. It adds no fields or methods.

This suggests planned restaurant support that has not been implemented yet.

### `src/joi_delivery/domain/grocery_product.py`
Grocery-specific product model.

#### Classes

##### `GroceryProduct(Product)`
Dataclass for inventory items sold by grocery stores.

Additional fields:

- `weight: float`
- `threshold: int`
- `available_stock: int`
- `store: GroceryStore | None = None`
- `selling_price: float | None = None`
- `expiry_date: int | None = None`
- `discount: float | None = None`

###### `to_json(self)`
Extends `Product.to_json()` with grocery-specific fields plus:

- `store`: serialized nested store, if present
- `product_type: "grocery"`

Note:
If the store serializes its inventory and the inventory contains the same product, a recursive representation would be possible. In the current seed data, store inventories are empty, so that recursion does not occur.

### `src/joi_delivery/domain/outlet.py`
Base model for a sellable location.

#### Classes

##### `Outlet`
Dataclass with:

- `name: str`
- `outlet_id: str`
- `description: str | None = None`

###### `to_json(self)`
Returns a dictionary with `name`, `description`, and `outlet_id`.

### `src/joi_delivery/domain/grocery_store.py`
Grocery-store specialization.

#### Classes

##### `GroceryStore(Outlet)`
Dataclass extending `Outlet`.

Additional fields:

- `inventory: set[GroceryProduct] = field(default_factory=set)`

###### `to_json(self)`
Builds on `Outlet.to_json()` and adds an `inventory` list by serializing each product.

Design note:
The `inventory` field is not populated by the current seed generator, so serialized stores usually show an empty inventory list.

### `src/joi_delivery/domain/restaurant.py`
Restaurant specialization.

#### Classes

##### `Restaurant(Outlet)`
Placeholder subclass of `Outlet`. It adds no fields or methods.

### `src/joi_delivery/domain/user.py`
User model.

#### Classes

##### `User`
Dataclass representing a customer.

Fields:

- `user_id: str`
- `first_name: str`
- `last_name: str`
- `email: str`
- `phone_number: str`
- `cart: Cart | None = None`
- `username: str | None = None`

###### `to_json(self)`
Serializes the user, including:

- empty string for `username` when it is `None`
- nested cart JSON when `self.cart` exists

The `cart` link is not actively maintained by the current services; carts instead hold the user reference.

### `src/joi_delivery/domain/cart.py`
Shopping-cart model.

#### Classes

##### `Cart`
Dataclass with:

- `cart_id: str`
- `outlet: Outlet | None`
- `products: list[Product] = field(default_factory=list)`
- `user: User | None = None`

###### `to_json(self)`
Serializes:

- the cart ID
- nested outlet JSON
- a JSON list of products
- nested user JSON

This object is returned directly by `GET /cart/view`.

## Seed-data generator

### `src/joi_delivery/generator/__init__.py`
Package marker for generator modules. No runtime logic.

### `src/joi_delivery/generator/app_initializer.py`
Creates static in-memory data used at startup.

#### Classes

##### `SeedData`
Namespace-style helper containing static factory methods.

###### `create_cart_for_user(user_id: str, first_name: str, last_name: str, cart_id: str, outlet: GroceryStore, user: User) -> Cart`
Returns a `Cart` for the supplied `cart_id`, `outlet`, and `user`.

Observation:
`user_id`, `first_name`, and `last_name` are accepted but not used when constructing the cart.

###### `create_store(outlet_name: str, store_id: str) -> GroceryStore`
Creates a `GroceryStore` with `name=outlet_name` and `outlet_id=store_id`.

###### `create_user(user_id: str, first_name: str, last_name: str) -> User`
Creates a `User` and derives:

- `email` as `{first_name}.{last_name}@gmail.com`
- `phone_number` from `get_random_number_using_next_int()`

###### `get_random_number_using_next_int(min_val: int, max_val: int) -> int`
Returns a random integer via `random.randint(min_val, max_val)`.

###### `create_grocery_product(product_name: str, product_id: str, store: GroceryStore) -> GroceryProduct`
Creates a `GroceryProduct` with static defaults:

- `mrp=10.5`
- `weight=500.00`
- `threshold=10`
- `available_stock=30`
- `store=store`

#### Functions

##### `initialize_data()`
Logs startup messages and returns a dictionary containing:

- `users`
- `grocery_products`
- `stores`
- `cart_for_users`

#### Module-level seed objects

The module creates seed objects eagerly:

- `store101 = Fresh Picks`
- `store102 = Natural Choice`
- `user101 = John Doe`
- `cart_for_users` with entries for `user101` and `user102`
- `grocery_products` with `product101`, `product102`, and `product103`
- `users = [user101]`
- `stores = [store101, store102]`

Important inconsistency:
`cart_for_users` contains `user102`, but `users` contains only `user101`. As a result, user `user102` has a cart in memory but cannot be resolved through `UserService`.

## Known design limitations

### Missing error handling
The cart flow does not convert missing users, carts, or products into HTTP errors. Several paths can currently fail with `AttributeError` or `NoneType` issues.

### Inventory feature is only stubbed
The inventory-health endpoint ignores the store ID and returns only an empty 200 response.

### In-memory state only
Changes exist only for the life of the process. Restarting the application resets all carts and seed data.

### Mixed serialization strategy
The code combines:

- dataclass domain models
- manual `to_json()` methods
- Pydantic request/response models

This is acceptable for a kata-sized service but would usually be normalized in a larger application.

### Placeholder restaurant support
`FoodProduct` and `Restaurant` exist, but the rest of the codebase does not use them yet.

## Summary
This repository is intentionally small, but it is structured like a conventional backend:

- `main.py` wires the application together
- controllers expose HTTP behavior
- services hold the business logic
- domain dataclasses model carts, users, stores, and products
- generator code supplies static sample data
- tests currently cover the basic controller happy paths

The most important production gaps are inventory behavior, error handling, and persistence.
